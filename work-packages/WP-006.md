# WP-006

Title

Expert Engine

Status

In Progress

Priority

Critical

## Business Goal

Implement the APOS Expert Engine: the subsystem responsible for answering user
questions using the project's approved knowledge sources. It never invents
knowledge, never answers from assumptions, and never uses user conversations
as a knowledge source — conversations are stored only for future analysis.

---

## Scope

Create `packages/core/src/expert/` modules: `ExpertEngine`, `Conversation`,
`ConversationMessage`, `ConversationContext`, `ConversationRepository`,
`ConversationSession`, `ExpertResponse`, `Citation`, and an `index.ts` barrel.

Export the complete Expert Engine from `packages/core/src/index.ts`.

---

## Out of Scope

LLM implementation.

Document parsing.

Conversation analysis / learning.

Content generation.

Deployment logic.

Healthcare-specific or any vertical-specific logic.

Web retrieval (Approved External citations are architecture only).

---

## Dependencies

`docs/KERNEL/KNOWLEDGE_SYSTEM.md` (Knowledge Priority, Conversation Rule).

APOS Engineering role documentation.

`packages/core` from WP-003 (Core Engine foundation, models/ports) and WP-005
(Knowledge Acquisition Engine — `KnowledgeManager`, `KnowledgeRegistry`,
`KnowledgeIndex`, `SourcePriority`).

## Deliverables

- `packages/core/src/expert/Citation.ts` — `Citation` model referencing
  Project Knowledge, Vertical Pack, or Approved External sources, with
  priority derived structurally from `SourcePriority`.

- `packages/core/src/expert/ConversationMessage.ts` — a single message
  (`user` | `expert`) with timestamp and optional metadata.

- `packages/core/src/expert/Conversation.ts` — the complete, immutable
  conversation record (messages, timestamps, metadata, lifecycle state).

- `packages/core/src/expert/ConversationContext.ts` — a bounded, read-only
  view of recent messages passed to the Expert Engine; never a knowledge
  source.

- `packages/core/src/expert/ConversationSession.ts` — stateful class managing
  one in-progress conversation (add messages, close, snapshot to
  `Conversation`).

- `packages/core/src/expert/ConversationRepository.ts` — `InMemoryConversationRepository`
  storing only completed (`closed`) conversations; no analysis, no learning,
  no content generation.

- `packages/core/src/expert/ExpertResponse.ts` — answer, citations,
  confidence (clamped to [0, 1]), optional follow-up questions, metadata.

- `packages/core/src/expert/ExpertEngine.ts` — public interface. `answer(question,
  context, knowledgeManager)` returns an `ExpertResponse`. Defines
  `KnowledgeRetrievalPort` and `AnswerProviderPort` as the AI-provider seams;
  no concrete implementation of either ships in this foundation.

- `packages/core/src/expert/index.ts` — public barrel.

- Expert Engine exports added to `packages/core/src/index.ts`.

## Notes

Work package numbering: this task was requested as "WP-003", but WP-003 is
already recorded (In Progress) as the APOS Core Engine, and WP-004/WP-005 are
also taken (Object Model, Knowledge Acquisition Engine). To avoid overwriting
WP-003's history, this work is filed as WP-006 per user confirmation.

Honest default behavior instead of invented answers: `ExpertEngine.answer`
requires both a `KnowledgeRetrievalPort` (retrieval) and an `AnswerProviderPort`
(answer generation) to produce a real answer — neither has a concrete
implementation in this foundation, since real retrieval needs document
understanding and real answering needs an LLM. Without them (the only state
this build can ever be in), the engine returns a fixed, honest "insufficient
knowledge" response with zero citations and zero confidence, rather than
fabricating an answer. This is the concrete embodiment of "never invents
knowledge" and "never answers from assumptions" until an LLM-backed provider
is introduced.

`Conversation` vs `ConversationSession`: `Conversation` is the immutable, complete
record (data only); `ConversationSession` is the stateful class that manages
an in-progress conversation and produces a `Conversation` via `toConversation()`.
This mirrors the `Object`/`ObjectSnapshot` split from WP-004 (behavior vs. data).

`ConversationRepositoryPort.save` rejects conversations whose state is not
`closed`, enforcing "stores completed conversations" structurally rather than
by convention.

`Citation.priority` is derived structurally from `sourceType` via
`priorityForCitationSourceType`, reusing `SourcePriority` from WP-005 so
citations and the Knowledge System's four-tier model can never drift apart.
General Web Search is intentionally excluded from `CitationSourceType`, since
KNOWLEDGE_SYSTEM.md never treats it as "approved".

The Expert Engine takes the concrete `KnowledgeManager` (WP-005) as a
parameter rather than introducing a new port interface for it, since the
engine only calls its existing public methods (`getRegistry()`, `getIndex()`
indirectly via injected retrievers); no behavior of `KnowledgeManager` was
modified.
